PicoSearch is a lightweight fuzzy search library that uses an fzf-inspired algorithm to find and rank matches across object collections. The core design prioritizes speed, accuracy, and simplicity.

## Architecture overview

PicoSearch consists of three main components:

<Steps>
  <Step title="Fuzzy scoring algorithm">
    The heart of PicoSearch is a fuzzy matching algorithm (`fuzzyScore.ts:6`) that scores how well a pattern matches a target string. It uses a three-phase approach: forward scan, reverse scan, and scoring.
  </Step>
  
  <Step title="Multi-key search orchestration">
    The main search function (`pico.ts:20`) coordinates searches across multiple object keys, applying weights and combining scores to rank results.
  </Step>
  
  <Step title="Weighted scoring">
    A utility function (`utils.ts:8`) calculates weighted averages when combining scores from multiple fields.
  </Step>
</Steps>

## Search flow

When you call `picoSearch()`, here's what happens:

### 1. Input normalization

The search term is normalized and split into tokens:

```typescript
const trimmed = searchTerm.trim().toLowerCase();
const terms = trimmed.split(/\s+/).filter(Boolean);
```

This allows PicoSearch to handle multi-word queries intelligently.

### 2. Key normalization

Keys can be specified as strings or objects with weights:

```typescript
const normalizedKeys = keys.map((k) =>
  typeof k === "string" ? { name: k, weight: 1 } : k,
);
```

This enables you to prioritize certain fields (e.g., title over description).

### 3. Scoring strategy

PicoSearch uses different strategies based on query type:

<Accordion title="Single-term queries">
  For single words, each key is scored independently and combined using a weighted average:

  ```typescript
  const scores = values.map((v) => (v ? fuzzyScore(v, terms[0]) : 0));
  const weights = normalizedKeys.map((k) => k.weight);
  const similarity = weightedAverage(scores, weights);
  ```

  This produces a smooth ranking where matches across high-weight fields score better.
</Accordion>

<Accordion title="Multi-term queries">
  For multiple words, each term must match somewhere. The final score is the minimum across all terms:

  ```typescript
  const termScores = terms.map((term) => {
    let best = 0;
    for (let i = 0; i < values.length; i++) {
      if (values[i]) {
        const s = fuzzyScore(values[i], term);
        if (s > best) best = s;
      }
    }
    return best;
  });

  const similarity = Math.min(...termScores);
  ```

  This ensures all query terms are represented in the result.
</Accordion>

### 4. Threshold filtering

Only results meeting the similarity threshold are included:

```typescript
const threshold = config?.threshold ?? 0.3;

if (similarity >= threshold) {
  results.push({ object: obj, similarity });
}
```

<Info>
  The default threshold of 0.3 provides a good balance between recall and precision. Lower values return more results; higher values are more strict.
</Info>

### 5. Result ranking

Finally, results are sorted by similarity score (highest first):

```typescript
results.sort((a, b) => b.similarity - a.similarity);
return results.map((r) => r.object);
```

## Key design principles

**Subsequence matching**: PicoSearch finds matches even when characters are non-consecutive. For example, "fzf" matches "fuzzyScore".

**Word boundary awareness**: Matches at word boundaries (camelCase, separators) score higher than mid-word matches.

**Length normalization**: Shorter targets are slightly preferred as tiebreakers, preventing long strings from dominating results.

**Configurable thresholds**: You control the minimum match quality through the threshold parameter.

<Note>
  The fuzzy scoring algorithm is inspired by fzf, the popular command-line fuzzy finder, but optimized for JavaScript and web use cases.
</Note>

## Performance characteristics

PicoSearch is designed for client-side search with these characteristics:

- **Time complexity**: O(n × m × k) where n is the number of objects, m is the average string length, and k is the number of keys
- **Space complexity**: O(n) for storing results
- **No indexing**: Searches run on-demand without preprocessing, making it ideal for dynamic datasets
- **Lightweight**: The core algorithm is ~70 lines of code with no dependencies

<Info>
  For datasets under 10,000 items, PicoSearch typically returns results in under 10ms on modern hardware.
</Info>
