### What makes it tiny

PicoSearch achieves its sub-1kb size through:

- **Single algorithm**: Uses only fuzzy scoring, no complex string distance calculations
- **Minimal API surface**: One function, one purpose
- **No dependencies**: All code is self-contained
- **Efficient implementation**: Greedy forward+reverse scan instead of dynamic programming
- **Zero runtime overhead**: No setup, no indexing, no preprocessing

### Algorithm complexity

<Accordion title="Time complexity">
  **O(n × m × k)** where:
  - `n` = number of items
  - `m` = average string length
  - `k` = number of searchable keys per item

  The fuzzy scoring algorithm itself is **O(m)** per string comparison using a greedy forward+reverse scan, not dynamic programming.

  For multi-word queries, multiply by the number of search terms.
</Accordion>

<Accordion title="Space complexity">
  **O(n)** for storing results array.

  No preprocessing or index building means zero memory overhead before search.
</Accordion>

## Optimization strategies

### Threshold tuning

The threshold parameter is the most powerful performance optimization:

```typescript
// Higher threshold = faster search (fewer results to sort)
const results = picoSearch(largeDataset, "query", ["name"], {
  threshold: 0.7, // Only high-quality matches
});

// Lower threshold = slower search (more results to process)
const results = picoSearch(largeDataset, "query", ["name"], {
  threshold: 0.1, // Include weak matches
});
```

<Tip>
Increasing the threshold from 0.3 to 0.6 can significantly reduce the number of results that need sorting, speeding up searches on large datasets.
</Tip>

### Limit searchable keys

Only search the fields you need:

```typescript
// Slower: searching many fields
picoSearch(data, query, ["name", "title", "description", "tags", "category"]);

// Faster: search only essential fields
picoSearch(data, query, ["name", "title"]);
```

### Debounce user input

For real-time search as users type, debounce to avoid unnecessary searches:

```typescript
import { picoSearch } from "@scmmishra/pico-search";

let timeout;
const searchInput = document.querySelector("#search");

searchInput.addEventListener("input", (e) => {
  clearTimeout(timeout);
  timeout = setTimeout(() => {
    const results = picoSearch(data, e.target.value, ["name"]);
    displayResults(results);
  }, 150); // Wait 150ms after user stops typing
});
```

### Slice results

If you only need the top N results, slice the array:

```typescript
const topResults = picoSearch(data, query, ["name"]).slice(0, 10);
```

Note: PicoSearch still scores all items, so this doesn't improve search speed, but it helps with rendering performance.

## Algorithm details

PicoSearch uses a greedy forward+reverse subsequence scan inspired by fzf:

### Scoring process

1. **Forward scan**: Find the first valid subsequence match
2. **Reverse scan**: Tighten the match to the smallest window
3. **Score calculation**: Reward consecutive runs, word boundaries, and prefix matches
4. **Length normalization**: Apply mild length penalty to prefer shorter targets

```typescript
// Example: searching "jd" in "John Doe"
// 1. Forward scan: finds J...D
// 2. Reverse scan: tightens to J...D (no improvement here)
// 3. Score bonuses:
//    - Word boundary match (J at start, D after space)
//    - Short target length
//    - Result: high score
```

### Why this algorithm

<CardGroup cols={2}>
  <Card title="Fast" icon="rocket">
    Greedy scan is O(m) instead of O(m²) dynamic programming used in edit distance.
  </Card>
  <Card title="Intuitive" icon="lightbulb">
    Matches user expectations from tools like fzf, VSCode, and Sublime Text.
  </Card>
  <Card title="Compact" icon="compress">
    Simple algorithm means less code and smaller bundle size.
  </Card>
  <Card title="Predictable" icon="check">
    Consistent ranking: prefix > word boundary > contiguous > scattered matches.
  </Card>
</CardGroup>
